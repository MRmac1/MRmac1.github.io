React简历制作 --- webpack + React开发环境搭建

发表于 2016-04-27   |   分类于 React   |  

此篇为制作React简历小项目系列开山篇

前言

经过nodeJS自2009年被天才程序员Ryan Dahl开源以来，JS这项古老的前端技术便在服务器端野蛮的发展。适逢CommonJS标准的释出，node实现了commonJS的大部分标准，非常完美的适应了其在后端的发展。

npm一直被标榜为node的秘密武器，确实，仓库里有25万的开源仓库，对js开发者是一笔巨大的财富了。在写后端代码时候，你只需要在本地npm install repo，然后在代码里require(‘repo’)就可以完成。确实是非常的舒服。

JS前端在前端发展又怎么样呢？在前端的发展也是风起云涌，前端页面质量要求越来越高，前端项目再也不是简简单单的HTML+jQuery+css便可以一劳永逸的了。界面的不断革新会导致前端实际上的工作量比后端要大。这时候emberJS，angularJS和React的相继出现导致前端开发的大变革。但是前端JS和后端JS始终是分置在HTTP的两端，但是前面介绍过的那么多npm模块，我们可不可以拿过来直接用呢？答案是肯定的，因为都是JS语言，并不存在兼容性问题，问题的关键在于模块加载方式。

那我们就来看看前端的模块加载方式。

前端模块加载方式

  1. script标签
  2. CMD
  3. AMD
  4. ES6模块import机制
script标签加载

script标签无疑是最早也是使用最广的‘类模块’加载语言，为什么我要用‘类’呢？原因是因为script标签加载的缺点。使用标签进行加载，会导致引入的js文件里很容易出现全局对象污染。有时候引入一两个不太严谨的js文件，里面的命名和你本地的冲突就会导致一些非常奇怪的bug。标签引入还有点非常不好，就是加载顺序很重要，而且是同步式执行。这对浏览器天生异步的特性是相违背的。顺序加载意味着你需要厘清楚各js文件之间的依赖关系。噢，这可是项大工程！

CMD和AMD的撕逼

CommonJS规范是node在后端采用的规范，那其放在前端是否适用呢？结果是显然易见的，同步式加载和浏览器异步特性不符，大家都有共识，但是模块何时导入这个问题上。曾经引起了前端的大争论，直接导致前端发展为AMD方阵和CMD方阵。其中细节就不多追究了。

简而言之，CMD是所有依赖都加载完全才会解析代码。而AMD是按需加载。

确实非常简洁，但是现在支持这种引入方式的模块还非常少。

小结下

就目前而言，AMD好像是最符合前端包管理机制的方法了，但是其也有缺点，就是用AMD方式加载模块的方式不明显。各种前端模块加载方式看起来都不是完美的，那么有那么一种又可以自由选择加载方式(CMD,AMD或者是ES6)的模块管理办法么？答案就是webpack。

webpack解决的问题

用户自由选择加载方式

这个问题在上面有提到过，真正选择使用什么语法的是用户，用户只需要关心我在使用模块前声明了模块引入，然后webpack就会帮我使用最优的方式进行模块引入就可以了。

传输方式的改变

传统方式传输,要么是一次加载一个模块,要么是一次加载所有的模块.webpack要做到一次请求加载可选择的加载模块，具体的可以参考下这篇文章

管理各种资源文件

除了js，你可以想象css和jep都是以模块方式加载么？webpack就可以做到这一点。引入css只需要require(/dist/index.css)即可。
